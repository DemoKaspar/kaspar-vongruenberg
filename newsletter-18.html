Newsletter Issue #18

Developer Self-Service

Hey {NAME},

Issue #18! Today I want to dive deep into something that's become the holy grail of platform engineering: developer self-service. Everyone talks about it, everyone wants it, but most teams are building it wrong.

The problem isn't technical - it's conceptual. We're building self-service tools for operations teams, not self-service experiences for developers.

What Self-Service Really Means

True developer self-service isn't about giving developers access to infrastructure tools. It's about giving them the capability to solve business problems without being blocked by infrastructure concerns.

Let me illustrate the difference:

Infrastructure self-service: "Here's a web form where you can configure a Kubernetes deployment with the right resource limits, health checks, and ingress rules."

Developer self-service: "Push your code to the main branch and it will be running in production in 5 minutes."

The first approach makes developers part-time operations engineers. The second approach makes infrastructure invisible to developers.

The Abstraction Level Problem

Most platform teams get the abstraction level wrong. They abstract away the syntax (YAML becomes forms, CLI commands become buttons) but not the concepts (developers still need to understand pods, services, ingress, etc.).

This creates the worst of both worlds: developers still need deep infrastructure knowledge, but they're forced to interact with it through a less flexible interface.

Good developer self-service abstracts away concepts, not just syntax. Developers should think in terms of applications, not infrastructure resources.

The Progressive Disclosure Principle

The best self-service platforms use progressive disclosure - they're simple by default but allow complexity when needed.

Level 1: "Deploy my application" (one button/command)

Level 2: "Deploy my application with specific performance requirements" (a few additional parameters)

Level 3: "Deploy my application with custom infrastructure configurations" (full access to underlying systems)

Most developers should never need Level 2 or 3. But when they do, the platform should provide a path without breaking the simple experience for everyone else.

Common Self-Service Anti-Patterns

Here are the patterns I see that prevent true developer self-service:

Configuration explosion: Adding more and more options instead of better defaults. If your deployment form has 50 fields, you've built infrastructure-as-a-service, not developer self-service.

Process digitization: Taking manual approval workflows and making them digital. Self-service that requires approval isn't self-service.

Tool consolidation: Building portals that expose multiple tools through a single interface. This reduces context switching but doesn't reduce complexity.

Expert-user design: Designing interfaces for power users instead of typical developers. If it takes training to use your self-service platform, it's not self-service.

The Right Mental Model

Instead of thinking "How can we make infrastructure easier to manage?", think "How can we eliminate infrastructure management?"

The goal isn't to teach developers to be operations engineers. The goal is to let developers focus on application logic while the platform handles operational concerns automatically.

This requires shifting from configuration-driven to intent-driven interfaces:

Configuration-driven: "Set CPU to 500m, memory to 1Gi, replicas to 3, enable health checks on port 8080..."

Intent-driven: "This is a web service that needs to handle moderate traffic"

Real-World Examples

Let me give you some concrete examples of good developer self-service:

Deployment: Instead of asking developers to configure container orchestration, ask them what kind of application they're deploying (web service, background worker, batch job) and handle the infrastructure concerns automatically.

Data storage: Instead of asking developers to configure database instances, ask them what kind of data they need to store (relational, cache, documents) and provision appropriate, well-configured resources.

Monitoring: Instead of asking developers to set up dashboards and alerts, automatically instrument applications and alert on standard SLIs (error rate, response time, throughput).

Environment management: Instead of asking developers to configure environment-specific settings, automatically promote code through environments with appropriate configurations.

The Platform-Application Contract

Good developer self-service requires a clear contract between the platform and applications:

Applications provide: Code, dependencies, and basic metadata about what they do

Platform provides: Deployment, scaling, monitoring, security, and operational concerns

The contract should be simple enough that any developer can understand it, but powerful enough to handle real-world complexity.

This is why workload specifications like Score are so important - they provide a simple way for applications to declare their needs without getting into infrastructure implementation details.

Measuring Self-Service Success

How do you know if your self-service platform is working?

Time to first deployment: How long from "I have code" to "it's running in production"? Should be minutes, not hours.

Support ticket volume: Are developers able to solve problems independently, or do they constantly need help?

Onboarding time: How long before new developers are productive? How much platform-specific knowledge do they need?

Adoption rate: Are developers choosing to use the platform, or being forced to use it?

Escape hatch usage: How often do developers need to work around the platform? Some usage is expected, but high usage indicates the platform isn't meeting developer needs.

The Self-Service Spectrum

Not everything needs to be self-service on day one. Start with the most common, well-understood use cases and gradually expand:

Phase 1: Basic deployment and environment management

Phase 2: Database and service provisioning

Phase 3: Advanced networking and security configurations

Phase 4: Custom resource types and platform extensions

The key is making each phase genuinely self-service, not just adding more configuration options.

Cultural Changes Required

Building true developer self-service requires cultural changes, not just technical ones:

From control to trust: Platform teams need to trust developers to use capabilities responsibly, instead of trying to control every decision.

From perfection to iteration: It's better to build simple self-service for common cases and iterate than to try to handle every edge case upfront.

From expert users to typical users: Design for the average developer, not the most sophisticated users.

From features to outcomes: Measure success based on developer productivity, not platform feature count.

The Long Game

Developer self-service is ultimately about leverage. When done right, it multiplies the effectiveness of both platform teams and development teams.

Platform teams can focus on building capabilities instead of responding to requests. Development teams can focus on business problems instead of infrastructure concerns.

But this only works if the self-service experience is genuinely better than the manual alternative. If your self-service platform is slower, more complex, or less reliable than manual processes, developers will find ways around it.

The Score Connection

This is exactly why we're building Score - to provide a simple, universal way for applications to express their needs without getting into infrastructure implementation details.

Score files let developers say "I need a database" instead of "I need a PostgreSQL instance with these specific configurations". The platform can then provide the appropriate database for each environment and context.

It's intent-driven configuration that enables true developer self-service.

Questions for Your Team

As you think about developer self-service in your organization:

  • Are your developers configuring infrastructure, or expressing application needs?
  • How much platform-specific knowledge do developers need to be productive?
  • Are you eliminating manual processes, or just digitizing them?
  • Can developers accomplish common tasks without help from the platform team?
  • Are you designing for typical developers, or power users?

The goal of developer self-service isn't to make infrastructure management easier - it's to eliminate the need for developers to manage infrastructure at all.

What's your experience with developer self-service? What works well in your organization? What doesn't?

Kaspar
