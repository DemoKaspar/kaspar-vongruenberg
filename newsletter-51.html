Newsletter Issue #51

A single file to rule them all - announcing Score!

Hey {NAME},

Today I'm excited to announce something we've been working on for months: Score, a workload specification that could change how we deploy applications.

But first, let me tell you why we built it.

The Configuration Hell We're All Living In

Every platform engineer I talk to is dealing with the same fundamental problem: applications need to be configured differently in different environments, but managing those configurations is a nightmare.

You start with a simple application. It needs a database and maybe a cache. In development, you use SQLite and skip the cache. In staging, you use a small Postgres instance and Redis. In production, you use a clustered Postgres with read replicas and a Redis cluster with failover.

How do you manage this complexity?

Most teams end up with environment-specific configuration files scattered across repositories. Docker Compose files for local development, Helm charts for production, different CI/CD configurations for each environment. Every time you add a new service dependency, you have to update multiple files in multiple places.

It's maintenance hell, and it scales terribly.

The Score Approach

Score takes a different approach. Instead of describing how to configure infrastructure in each environment, you describe what your application needs - its dependencies, its resource requirements, its external interfaces.

A Score file might look like this:

apiVersion: score.dev/v1b1
metadata:
  name: hello-world

containers:
  hello:
    image: nginx:latest
    variables:
      CONNECTION_STRING: ${resources.db.connection}

resources:
  db:
    type: postgres
  cache:
    type: redis
    properties:
      persist: true

That's it. No Kubernetes YAML, no environment-specific configuration, no infrastructure details. Just a description of what the application needs to run.

How It Works

The magic happens in the translation layer. A platform orchestrator (like Humanitec, or a custom tool) reads the Score file and translates it into environment-specific configuration based on policies and resource definitions.

In development, that `postgres` resource might become a local Docker container. In staging, it might become a managed database instance. In production, it might become a highly available database cluster with backups and monitoring.

The developer doesn't have to know or care about these differences. They just specify that they need a Postgres database, and the platform figures out how to provide it.

Why This Matters

Score solves several problems that have been bugging me for years:

Configuration drift: When configuration is environment-specific, it inevitably drifts. Development stops looking like production. Staging has different resource limits than production. Score keeps environments consistent by deriving all configuration from a single source of truth.

Platform adoption: It's hard to get developers to adopt platform abstractions when they have to learn platform-specific configuration formats. Score lets developers describe their applications in a simple, universal format.

Environment parity: Score makes it much easier to maintain parity between environments because the same workload specification is used everywhere.

Onboarding: New developers don't have to learn environment-specific deployment processes. They can focus on the application logic and let the platform handle the infrastructure concerns.

Platform Agnostic

Here's what I'm most excited about: Score isn't tied to any specific platform or tool. It's an open standard that can be implemented by anyone.

We're open-sourcing the specification and providing reference implementations for common platforms. If you're using Kubernetes directly, we'll have tools to translate Score files to Kubernetes manifests. If you're using Docker Compose for local development, we'll have tools for that too.

The goal is to create a universal workload specification that works across different platforms and tools, reducing vendor lock-in and increasing portability.

Beyond Infrastructure

While Score started as a way to solve infrastructure configuration problems, I think it has broader implications for how we think about application deployment.

Score files become a form of documentation - they clearly describe what an application needs to run. They become a contract between application developers and platform teams. They become a foundation for tooling, automation, and policy enforcement.

I can imagine Score files being used for:

  • Cost estimation - analyzing resource requirements before deployment
  • Security scanning - checking for compliance violations in application dependencies
  • Capacity planning - understanding aggregate resource requirements across applications
  • Testing - ensuring that applications work correctly with different resource configurations

Getting Started

Score is available now at score.dev. We've got documentation, examples, and tools for getting started.

If you're a platform engineer, I'd love to hear your thoughts on the specification. If you're an application developer, I'd love to see how Score fits into your deployment workflow.

If you're a tool builder, let's talk about how Score could integrate with your platform. The more tools that support Score, the more valuable it becomes for everyone.

The Long Game

I believe Score represents a fundamental shift in how we think about application deployment. Instead of applications being responsible for infrastructure configuration, they focus on describing their requirements and let platforms handle the implementation details.

This separation of concerns is what makes scalable platform engineering possible. Applications become more portable, platforms become more flexible, and everyone spends less time fighting with configuration files.

But Score is just the beginning. The real work happens in building the tooling ecosystem around it, integrating it into existing platforms, and proving that this approach works at scale.

That's the work I'm most excited about for the next few months.

What do you think? Does Score solve problems you've been facing? How would it fit into your platform?

Let me know - I'd love to hear your thoughts.

Kaspar
