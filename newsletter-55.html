Newsletter Issue #55

You're getting developer self-service wrong. Here's why

Hey {NAME},

Issue #55, and today I want to talk about something that's become a holy grail in platform engineering: developer self-service. Everyone wants it, everyone's building it, but most teams are getting it fundamentally wrong.

The problem isn't technical - it's conceptual. We're building self-service interfaces for operations teams, not self-service experiences for developers.

The Self-Service Illusion

Here's what most "developer self-service" looks like in practice:

A developer wants to deploy a new service. They open the internal developer portal, navigate through multiple forms asking about:

  • CPU and memory requirements
  • Database instance types and sizes
  • Load balancer configurations
  • Network security groups and rules
  • Monitoring and alerting thresholds
  • Backup and disaster recovery settings

After 20 minutes of configuration and three approval workflows, they get their infrastructure. We call this "self-service," but it's really just "self-administered operations complexity."

True developer self-service would be: "I want to deploy my application." Click. Done.

The Operations-Centric Mistake

Most platform teams build self-service by exposing infrastructure concepts through prettier interfaces. Instead of using Terraform directly, developers use a web form that generates Terraform. Instead of manually configuring load balancers, they use a portal that configures load balancers.

This approach makes operations teams feel good because they've "automated" their processes. But it doesn't make developers' lives better - it just makes them part-time operations engineers.

The fundamental mistake: optimizing for infrastructure flexibility instead of developer productivity.

What Developers Actually Want

When developers say they want self-service, they don't mean they want to configure infrastructure. They mean they want to solve business problems without being blocked by infrastructure concerns.

They want to:

  • Deploy code quickly and safely - without understanding container orchestration
  • Store and query data - without becoming database administrators
  • Handle traffic spikes - without understanding load balancing algorithms
  • Integrate with other services - without configuring service meshes
  • Debug problems in production - without setting up monitoring dashboards

Notice the pattern? They want capabilities, not configurations.

Application-Centric Self-Service

True developer self-service starts with application requirements, not infrastructure resources.

Instead of asking "What kind of database do you want?", ask "What does your application need to store?"

Instead of asking "How many CPU cores?", ask "What type of workload is this?"

Instead of asking "Which monitoring tools?", ask "How do you want to be notified when things break?"

The platform should translate application needs into infrastructure configurations automatically, using sensible defaults and organizational policies.

The Score Approach

This is exactly why we built Score the way we did. A Score file describes what an application needs, not how infrastructure should be configured:

resources:
  database:
    type: postgres
  cache: 
    type: redis
    properties:
      persist: true

containers:
  web:
    image: myapp:latest
    variables:
      DATABASE_URL: ${resources.database.connection_string}

That's it. No CPU limits, no network configurations, no backup policies. Just "I need a database and a cache."

The platform figures out how to provide those resources appropriately for each environment based on policies, not developer decisions.

Progressive Disclosure

Good developer self-service uses progressive disclosure - simple by default, complex when necessary.

Level 1: "I want to deploy a web application"

Level 2: "I want to deploy a web application with specific performance characteristics"

Level 3: "I want to deploy a web application with custom infrastructure configurations"

Most developers should never need to go beyond Level 1. Some will occasionally need Level 2. Very few will need Level 3.

But most platforms start at Level 3 and wonder why adoption is low.

The Right Abstractions

The key to good developer self-service is finding the right level of abstraction. Too high, and you can't handle real-world complexity. Too low, and you're just infrastructure-as-a-service with better UX.

Good abstractions are:

Intentional: They map to how developers think about their applications, not how operations teams think about infrastructure.

Composable: Simple concepts can be combined to handle complex scenarios without exposing that complexity to simple scenarios.

Extensible: When developers do need more control, they can get it without breaking the abstraction for everyone else.

Reversible: Teams can start simple and add complexity over time, or start complex and simplify as patterns emerge.

Measuring True Self-Service

How do you know if you're providing real developer self-service?

Time to first deployment: How long from "I have code" to "it's running in production"? Should be minutes, not hours.

Infrastructure knowledge required: Can a developer be productive without understanding Kubernetes, load balancers, or database tuning?

Support ticket volume: Are developers able to solve common problems without help?

Developer satisfaction: Do developers feel empowered or burdened by the self-service tools?

Adoption rate: Are teams choosing to use the platform, or being forced to?

Common Anti-Patterns

Here are some patterns I see that prevent true developer self-service:

Configuration proliferation: Adding more and more configuration options instead of better defaults.

Infrastructure exposure: Making developers choose between 15 different instance types instead of 3 workload patterns.

Process automation: Digitizing approval workflows instead of eliminating the need for approvals.

Tool consolidation: Building single interfaces that expose multiple underlying tools instead of unified experiences.

Expert-driven design: Designing interfaces based on what power users need instead of what typical developers need.

The Cultural Shift

Building true developer self-service requires a cultural shift in platform teams:

From: "How can we make infrastructure easier to configure?"
To: "How can we eliminate infrastructure configuration?"

From: "Developers should understand the infrastructure they're using"
To: "Infrastructure should be invisible to developers by default"

From: "We need to expose all the powerful features"
To: "We need to hide complexity behind sensible defaults"

This shift is hard for platform teams because it requires giving up control and trusting that abstractions will handle most use cases correctly.

The Payoff

When you get developer self-service right, the benefits are enormous:

Higher developer velocity: Teams can focus on application logic instead of infrastructure concerns.

Better operational consistency: When the platform handles infrastructure decisions, they're made consistently and according to organizational policies.

Reduced support burden: Fewer infrastructure decisions mean fewer things that can go wrong and fewer things developers need help with.

Easier onboarding: New developers can be productive immediately without learning your infrastructure stack.

Platform team leverage: Instead of responding to requests, the platform team can focus on improving capabilities and abstractions.

Getting Started

If you want to move toward true developer self-service:

Start with user research: What do developers in your organization actually struggle with? Don't assume - ask.

Map developer journeys: From idea to production, what steps do developers take? Where do they get stuck?

Identify common patterns: What do most applications need? Build great defaults for the 80% case.

Hide infrastructure by default: Developers should interact with application concepts, not infrastructure concepts.

Measure developer experience: Survey developers regularly about their platform experience. Their opinion matters more than yours.

The goal isn't to make infrastructure configuration easier - it's to make infrastructure configuration unnecessary for most developers most of the time.

What's your experience with developer self-service? Are your developers configuring infrastructure or capabilities?

Kaspar
