Newsletter Issue #79

Reference Architectures for GitOps based IDPs

Hey {NAME},

I've been getting a lot of questions lately about reference architectures for GitOps-based Internal Developer Platforms. People want to understand how all the pieces fit together in practice.

So I thought I'd walk you through a few different patterns we see working well in production, and explain when you might choose one over another.

Pattern 1: The Single Source of Truth

This is probably the most common pattern we see:

  • Application repos contain Score files (or similar workload specifications)
  • A single "platform config" repo contains all environment-specific configurations
  • A platform orchestrator (like Humanitec) reads from app repos, matches configurations from the platform repo, and generates final Kubernetes manifests
  • These manifests get written to a "gitops repo" that ArgoCD (or Flux) watches

The big advantage here is simplicity - there's one place where all your platform configurations live, making it easy to maintain consistency and implement governance.

The downside is that as you scale, the platform repo can become a bottleneck. Every team needs to touch it to add new configurations or modify existing ones.

Pattern 2: Federated Configuration

This pattern distributes configuration ownership:

  • Each team has their own "team platform repo" with their specific configurations
  • A central "base platform repo" contains shared/default configurations
  • The platform orchestrator reads from all these repos and merges configurations based on precedence rules
  • Final manifests still get generated to a centralized GitOps repo

This pattern scales much better because teams can self-serve on configurations without creating bottlenecks. But it's more complex to set up and requires good governance to prevent teams from shooting themselves in the foot.

Pattern 3: Environment-Specific GitOps

Some organizations prefer to have separate GitOps repos per environment:

  • Application repos still contain workload specs
  • Platform configurations are environment-specific
  • Separate GitOps repos for dev, staging, production
  • Different promotion processes between environments

This pattern gives you maximum isolation between environments, which is great for security and compliance. But it can make it harder to ensure consistency across environments.

Choosing the right pattern

Your choice depends on a few factors:

Team size and structure: If you have a small number of teams and centralized platform management, Pattern 1 works great. As you grow and need more team autonomy, Pattern 2 becomes more attractive.

Compliance requirements: If you need strong environment isolation for regulatory reasons, Pattern 3 might be required.

Platform team capacity: Pattern 1 requires the least setup but creates the most ongoing work for platform teams. Pattern 2 requires more sophisticated tooling but scales better.

Common pitfalls to avoid

Whatever pattern you choose, here are some things to watch out for:

  • Don't skip the workload specification layer. Going directly from app repos to GitOps repos without an abstraction layer makes it impossible to maintain environment consistency.
  • Don't put environment-specific configuration in application repos. This defeats the purpose of having a platform and makes standardization impossible.
  • Don't forget about secret management. Your GitOps repos shouldn't contain actual secrets - use tools like External Secrets Operator or Sealed Secrets.
  • Don't ignore RBAC. Make sure your GitOps repos have proper access controls so teams can't modify each other's configurations.

Tooling recommendations

For the platform orchestration layer, you need something that can read workload specs, apply configurations, and generate GitOps-ready manifests. Obviously I'm biased toward Humanitec, but there are other options like Backstage with the right plugins, or custom tooling built around Helm.

For GitOps, ArgoCD and Flux are both solid choices. ArgoCD has a nicer UI and better RBAC, Flux is more Kubernetes-native and has better Helm support.

For workload specifications, Score is obviously my preferred choice, but you could also use Helm values files or custom YAML schemas.

The key is making sure all these pieces work together smoothly and that your developers have a good experience throughout.

What pattern are you using? I'd love to hear about your experiences!

Kaspar
