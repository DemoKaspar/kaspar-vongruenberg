Newsletter Issue #89

The three types of platform users

Hey {NAME},

Not all developers use platforms the same way. Understanding the different types of platform users is crucial for building experiences that work for everyone.

After working with hundreds of developers across different organizations, I've identified three distinct platform user types. Each has different needs, different pain points, and different success metrics.

Here's how to design for all three.

Type 1: The Golden Path User

Who they are: Developers who want to ship features fast without worrying about infrastructure details. They're happy to follow established patterns if those patterns work well.

What they want:

  • Simple, reliable workflows
  • Clear documentation and examples
  • Minimal configuration required
  • Fast feedback loops
  • Someone else to handle the complexity

What frustrates them:

  • Too many options and configurations
  • Unclear error messages
  • Slow or unreliable platform behavior
  • Having to learn infrastructure concepts

How to design for them:

  • Make your golden paths incredibly smooth
  • Provide sensible defaults for everything
  • Hide complexity behind simple interfaces
  • Invest heavily in clear, example-driven documentation
  • Make common operations one-click or one-command

Success metrics: Time to first deployment, deployment frequency, user satisfaction scores

Type 2: The Power User

Who they are: Experienced developers who understand infrastructure and want control. They use platforms to be more efficient, not to avoid complexity.

What they want:

  • Powerful APIs and CLI tools
  • Ability to customize and extend platform capabilities
  • Visibility into what the platform is doing
  • Escape hatches for edge cases
  • Integration with their preferred tools

What frustrates them:

  • Overly simplified interfaces that hide important details
  • Lack of customization options
  • Platform making decisions they want to control
  • Poor API design or documentation
  • No way to handle their specific use cases

How to design for them:

  • Provide comprehensive APIs alongside web interfaces
  • Offer advanced configuration options
  • Build in extensibility and plugin systems
  • Provide detailed logging and debugging tools
  • Create escape hatches for unsupported use cases

Success metrics: API usage, custom configuration adoption, feature request fulfillment

Type 3: The Reluctant User

Who they are: Developers who are required to use the platform but would prefer to manage infrastructure themselves. Often senior engineers with strong opinions about tooling.

What they want:

  • The platform to not get in their way
  • Transparency about what the platform is doing
  • Ability to override platform decisions
  • Migration paths away from the platform if needed
  • Proof that the platform adds value, not just restrictions

What frustrates them:

  • Forced adoption without clear benefits
  • Platform limitations that prevent their preferred approaches
  • Black box behavior they can't debug or understand
  • Performance or reliability issues
  • Platform team dismissing their concerns

How to design for them:

  • Be transparent about platform behavior and decisions
  • Provide detailed monitoring and debugging capabilities
  • Offer escape hatches and override mechanisms
  • Engage them in platform design discussions
  • Focus on proving value rather than enforcing compliance

Success metrics: Complaint reduction, voluntary adoption of new features, engagement in platform discussions

Designing for All Three Types

The challenge is building a platform that serves all three user types without compromising any of them. Here's how:

Layered interfaces

Provide multiple ways to interact with your platform:

  • Web UI: Simple workflows for golden path users
  • CLI tools: Powerful scripting for power users
  • APIs: Full programmatic access for advanced use cases
  • Configuration files: Declarative definitions for reluctant users who want to see everything

Progressive disclosure

Start simple, reveal complexity as needed:

  • Simple forms with "Advanced options" sections
  • Sensible defaults with override capabilities
  • Basic and advanced documentation tracks

Transparency options

Let users choose their level of visibility:

  • High-level status for golden path users
  • Detailed logs and metrics for power users
  • Raw configuration and debug info for reluctant users

Common Design Mistakes

Designing only for golden path users

Results in platforms that are too simple for complex use cases, driving power users and reluctant users to build workarounds.

Designing only for power users

Creates platforms that are too complex for most developers, leading to low adoption and lots of support requests.

Ignoring reluctant users

Reluctant users often become platform advocates if you win them over, or platform saboteurs if you don't. They're worth the investment.

Trying to serve everyone with the same interface

One-size-fits-all interfaces usually fit nobody well. Different user types need different interaction models.

User Type Evolution

Remember that user types can change:

  • Golden path users may become power users as they gain experience
  • Reluctant users may become advocates if they see clear value
  • Power users may prefer golden paths for routine tasks

Design your platform to support this evolution, not lock people into rigid categories.

Measuring Success Across User Types

Track different metrics for different user types:

Golden Path Users: Onboarding time, task completion rates, satisfaction scores

Power Users: API adoption, advanced feature usage, integration volume

Reluctant Users: Issue escalation rates, workaround usage, feedback sentiment

The Goal: Something for Everyone

The best platforms don't force all users into the same mold. They provide different paths for different needs while maintaining consistency and reliability.

Golden path users get simplicity. Power users get flexibility. Reluctant users get transparency.

Everyone gets value.
