Newsletter Issue #90

How to design platform APIs that developers actually want to use

Hey {NAME},

Platform APIs are the hidden foundation of developer experience. Get them right, and your platform feels intuitive and powerful. Get them wrong, and developers will build workarounds.

I've reviewed dozens of platform APIs over the years. Here are the patterns that separate the good from the great.

API Design Philosophy

Design for the 90% use case

Your API should make common tasks trivial and complex tasks possible. Don't optimize for edge cases at the expense of everyday workflows.

Progressive disclosure

Start simple, then expose complexity as needed. New users should be able to accomplish basic tasks with minimal configuration.

Principle of least surprise

Developers should be able to guess how your API works based on conventions they already know.

Practical API Design Patterns

Resource-oriented design

Structure your APIs around resources (applications, environments, databases) rather than actions. This makes APIs more predictable:

Good:

  • GET /applications/{id}
  • POST /applications
  • PUT /applications/{id}/scale

Not so good:

  • GET /getApplication?id=123
  • POST /createApplication
  • POST /scaleApplication

Consistent naming conventions

Pick naming conventions and stick to them across all endpoints:

  • Use kebab-case for URLs: /deployment-configs
  • Use camelCase for JSON fields: deploymentConfig
  • Use consistent terminology: always "application" not sometimes "app"

Meaningful HTTP status codes

Use status codes correctly:

  • 200: Success
  • 201: Created successfully
  • 400: Client error (bad request format)
  • 401: Authentication required
  • 403: Forbidden (authenticated but not authorized)
  • 404: Not found
  • 409: Conflict (resource already exists)
  • 500: Internal server error

Developer Experience Priorities

Idempotency

Make operations idempotent where possible. Running the same API call multiple times should have the same result:

Good: PUT /applications/my-app/config (updating configuration)

Requires care: POST /applications/my-app/restart (restarting multiple times)

Batch operations

Provide batch endpoints for operations that developers commonly need to do in bulk:

  • POST /applications/batch (create multiple applications)
  • PUT /applications/batch/scale (scale multiple applications)

Async operations with status tracking

For long-running operations, return immediately with a status endpoint:

POST /applications/{id}/deploy
Returns: 202 Accepted
{
  "operationId": "deploy-123",
  "status": "pending",
  "statusUrl": "/operations/deploy-123"
}

Error Handling That Doesn't Suck

Structured error responses

Provide consistent error response format:

{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Deployment configuration is invalid",
    "details": [
      {
        "field": "replicas",
        "message": "Must be between 1 and 100"
      }
    ]
  }
}

Actionable error messages

Bad: "Invalid configuration"

Good: "Memory limit '512MB' is below the minimum required '1GB' for this application type"

Error codes for programmatic handling

Include machine-readable error codes so developers can handle specific errors programmatically.

Documentation That Developers Actually Use

OpenAPI specifications

Provide complete OpenAPI (Swagger) specs. This enables:

  • Interactive API documentation
  • Automated client generation
  • API testing tools
  • Contract testing

Runnable examples

Every API endpoint should have working code examples in multiple languages:

# Create application
curl -X POST /api/v1/applications \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my-app",
    "image": "nginx:latest",
    "replicas": 3
  }'

Use case-driven documentation

Organize documentation around what developers want to accomplish, not just listing all endpoints:

  • "Deploying your first application"
  • "Setting up automated deployments"
  • "Managing application configuration"
  • "Debugging deployment failures"

Versioning Strategy

URL-based versioning

Include version in the URL for clarity:

  • /api/v1/applications
  • /api/v2/applications

Backwards compatibility

Maintain backwards compatibility within major versions:

  • Add new fields, don't remove old ones
  • Add new endpoints, don't break existing ones
  • Provide migration paths for breaking changes

Deprecation process

When you need to break compatibility:

  1. Announce deprecation with 6+ months notice
  2. Add deprecation warnings to responses
  3. Provide migration documentation
  4. Support old and new versions concurrently
  5. Remove old version only after migration period

Authentication and Authorization

Clear authentication model

Make authentication requirements obvious:

  • Use standard methods (API keys, OAuth, JWT)
  • Document authentication clearly
  • Provide clear error messages for auth failures

Scope-based permissions

Use scoped permissions that align with how teams actually work:

  • applications:read (can view applications)
  • applications:write (can create/update applications)
  • applications:deploy (can deploy applications)

Performance and Reliability

Pagination for list endpoints

Always paginate list responses:

{
  "items": [...],
  "pagination": {
    "total": 156,
    "page": 1,
    "pageSize": 20,
    "hasNext": true,
    "nextUrl": "/applications?page=2"
  }
}

Rate limiting

Implement and communicate rate limits:

  • Include rate limit headers in responses
  • Provide clear error messages when limits are exceeded
  • Offer different limits for different types of operations

Caching headers

Use appropriate caching headers to reduce unnecessary requests.

Testing Your API Design

User journey testing

Test common user journeys end-to-end:

  • New developer creates their first application
  • Experienced developer sets up CI/CD integration
  • Team lead manages applications for their team

SDK development

Build SDKs for popular languages. This forces you to think about how your API feels to use programmatically.

Developer feedback sessions

Regular sessions where actual developers use your API and provide feedback on the experience.

The Goal: Invisible Excellence

The best platform APIs are invisible. Developers accomplish what they need without thinking about the API design.

They just work.

That's the standard to aim for.

Talk soon,

Kaspar
