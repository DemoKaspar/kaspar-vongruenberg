Newsletter Issue #96

Platform security: shift left, but not too far left

Hey {NAME},

"Security is everyone's responsibility" has become the de facto motto of modern software development. And while that's true in spirit, in practice it often means security becomes nobody's responsibility.

Platform engineering offers a better approach: centralized security implementation with distributed responsibility for security outcomes.

Let me explain.

The Problem with Traditional "Shift Left"

The traditional "shift left" security approach asks developers to:

  • Understand security scanning tools
  • Write secure code from scratch
  • Configure security policies correctly
  • Stay current on security threats
  • Remediate vulnerabilities in their applications

This works great if you have a team of security-minded senior engineers. But most organizations have developers with varying security expertise working under tight deadlines.

Result? Security theater instead of actual security.

The Platform Approach to Security

Instead of asking every developer to become a security expert, platform teams can:

Build security into the golden path

When developers use your platform's standard deployment pipeline, they automatically get:

  • Container scanning
  • Dependency vulnerability checks
  • Infrastructure security baselines
  • Network policies
  • Secrets management

Make insecure patterns harder than secure ones

Don't just provide security tools—make security the path of least resistance:

  • Pre-configured secure base images
  • Templates with security best practices baked in
  • Policy-as-code that prevents common misconfigurations
  • Automated certificate management

Provide actionable security feedback

Instead of dumping raw scanner output on developers, process security findings into actionable tasks:

  • "Update this dependency to version X.Y.Z"
  • "This endpoint needs authentication—here's how to add it"
  • "This container is running as root—use this base image instead"

The Three Layers of Platform Security

Layer 1: Infrastructure Security

Things developers should never have to think about:

  • Network segmentation
  • Host hardening
  • Infrastructure monitoring
  • Patch management

Layer 2: Application Security Defaults

Security that's built into templates and patterns:

  • Authentication and authorization frameworks
  • Secure communication defaults (TLS everywhere)
  • Input validation helpers
  • Secure configuration management

Layer 3: Security Visibility and Response

Tools that help developers understand and fix security issues:

  • Integrated security scanners
  • Security dashboards
  • Automated security testing
  • Incident response workflows

What Developers Still Own

This doesn't mean developers have no security responsibilities:

  • Business logic security: Authorization rules specific to their applications
  • Data handling: Proper handling of sensitive data within their code
  • Security hygiene: Responding to security findings and following secure coding practices
  • Threat modeling: Understanding the security implications of their architectural decisions

Making Security Invisible (The Good Kind)

The best platform security is invisible to developers. They get secure applications not because they manually configured security tools, but because security is built into the platform they're using.

This approach has several advantages:

  • Consistency: Security policies are applied uniformly across all applications
  • Expertise: Security specialists can focus on building great security infrastructure instead of training every developer
  • Speed: Developers can ship faster because they don't need to become security experts
  • Coverage: Security controls are applied to 100% of platform usage, not just teams that remember to use them

Implementation Tips

  1. Start with the infrastructure layer - Get the foundation right before worrying about application-level security
  2. Work with security teams - This isn't platform vs. security; it's platform enabling security
  3. Make secure patterns easy to find - Comprehensive documentation and examples
  4. Provide security training focused on platform usage - Teach developers how to use your security features, not how to build security from scratch
  5. Measure security outcomes, not just security activities - Focus on reducing actual vulnerabilities, not just increasing scanner usage

The Goal: Security by Default

The ultimate goal is "security by default"—applications that are secure not because developers made a hundred individual security decisions correctly, but because they're built on a secure platform.

Shift security left, but not all the way to individual developers. Shift it to the platform layer, where security experts can implement security once and apply it everywhere.

Talk soon,

Kaspar
