Newsletter Issue #97

Why your platform needs an API strategy

Hey {NAME},

Last week I was reviewing a platform architecture, and I asked a simple question: "How do teams interact with your platform programmatically?"

The answer: "Well, they can use our web interface, or... submit a ticket?"

This is a massive missed opportunity. Your platform's APIs aren't just technical implementation details—they're the foundation of how teams will actually use your platform.

APIs Are Your Platform's User Interface

Think about it: developers don't want to click through web forms to provision infrastructure. They want to include infrastructure provisioning in their CI/CD pipelines, their infrastructure as code, their automation scripts.

If your platform doesn't have a comprehensive API strategy, you're forcing teams to work around you instead of with you.

What Makes a Good Platform API?

1. RESTful and predictable

Follow REST conventions. Use standard HTTP status codes. Be consistent in your naming and response formats. Developers should be able to guess how your API works.

2. Comprehensive

Everything that can be done through your web interface should be possible via API. If teams can provision a database through your portal, they should be able to do it via API too.

3. Well-documented

OpenAPI specs, code examples, SDKs in popular languages. Your API documentation is your developer marketing—invest in it.

4. Idempotent

Running the same API call twice should have the same result as running it once. This makes automation much more reliable.

5. Properly versioned

Have a clear versioning strategy. Don't break existing integrations when you add new features.

The Three Layers of Platform APIs

I typically see successful platforms expose APIs at three levels:

Core Platform APIs

The fundamental building blocks: compute, storage, networking, databases. These are your "infrastructure primitives."

Service Template APIs

Higher-level abstractions: "create a web service," "spin up a batch job," "provision a data pipeline." These encode your golden paths.

Workflow APIs

End-to-end processes: "deploy this application," "create an environment," "run this migration." These represent complete user journeys.

API Strategy in Practice

Here's what I typically recommend:

Start with the golden paths

Don't try to API-fy everything at once. Start with your most common use cases and work outward.

Design APIs alongside UIs

Your web interface and APIs should be built together, not as an afterthought. Often, your web interface should just be a consumer of your APIs.

Provide multiple levels of abstraction

Some teams want low-level control, others want high-level convenience. Support both.

Invest in client libraries

SDKs in Go, Python, and JavaScript will dramatically improve adoption. Don't make teams write HTTP clients from scratch.

The Business Case for Platform APIs

Good platform APIs enable:

  • Self-service adoption - Teams can integrate with your platform without tickets or meetings
  • Automation - Infrastructure can be managed through CI/CD pipelines
  • Integration - Your platform can work with existing tools and workflows
  • Innovation - Teams can build new solutions on top of your platform

Common API Mistakes to Avoid

  • Exposing internal implementation details - Your API should be stable even when your implementation changes
  • Inconsistent error handling - Use standard HTTP status codes and consistent error message formats
  • No rate limiting - Protect your platform from runaway scripts
  • Ignoring authentication - Have a clear strategy for API keys, OAuth, or whatever auth mechanism fits your organization

The Bottom Line

Your platform's APIs determine how easily teams can adopt and integrate with your platform. Great APIs turn your platform into infrastructure that teams want to build on top of.

Bad APIs turn your platform into a bottleneck that teams work around.

Invest in your API strategy. Your adoption metrics will thank you.
